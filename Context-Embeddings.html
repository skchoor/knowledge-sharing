Context & Embeddings

Context and embeddings enable modern AI systems to better understand user intent and generate more coherent, relevant, and useful responses. They help bridge the gap between a user’s query and relevant knowledge, improving accuracy and personalization.

The journey from user query to AI answer.

User query (or attachment)
 │
 Chunking
 │
 Embedding model → vectors
 │
 Vector DB (store + ANN index)
 │
 Query embedding → top‑k nearest vectors
 │
 Prompt template (system + top_chunks + tools + user_query )
 │
 API call to LLM
 │
 Model response → post‑process
 │
 Answer shown to user
  1. Input & Chunking
    When a user submits a query or attachment, the context engine may slices the content into manageable chunks based on length constraints, document structure, or semantic boundaries, depending on the chunking strategy used. 
    content = "The quick brown fox jumps over the lazy dog. "

# `chunk_text(text, max_chars, overlap)` - e.g: ≤200 chars each, 40‑char overlap
chunks = chunk_text(content, max_chars=200, overlap=40)

# The overlapping region gives the embedding model enough surrounding words to preserve meaning.
  2. Embedding
    Each chunk is sent to an embedding model, which returns a dense vector ( a numerical array of floating-point values ) representing an abstraction of a data point in vector space. 
    ollama run qwen3-embedding "Exact date of CNY year 2026?"
# Returns a dense vector, e.g.:
[
  0.123456, -0.034567, 0.567890, -0.091234, 0.212345, -0.345678,
  0.456789, 0.098765, -0.876543, 0.234567, -0.345678, 0.456789,
  0.567890, -0.678901, 0.789012, -0.890123, 0.901234, -0.012345,
  ... (continues up to the full dimension, e.g., 2560 numbers)
]
  3. Storage & Indexing
    Vectors are written to a Vector DB alongside the raw chunk text (or a reference). The DB builds an Approximate Nearest-Neighbor (ANN) index over these entries, making them persistent and searchable. 
    import faiss, numpy as np
dim = len(vectors[0])   # dimension of qwen3‑embedding / 2560
index = faiss.IndexFlatL2(dim)
index.add(np.array(vectors).astype('float32'))
  4. Retrieval
    For a new query, the engine performs a top-K Nearest-Neighbor search, retrieving the most semantically similar chunks based on vector distance (e.g., cosine similarity). Because embeddings represent semantic meaning in a vector space, the retrieval process can identify conceptually related chunks even when different vocabulary is used. 
    query_vec = np.array(json.loads(embed(user_query)), dtype='float32')

# D=similarities, I=indices, k=neighbours
D, I = index.search(np.array([query_vec]).astype('float32'), k=5)
top_chunks = [chunks[i] for i in I[0]]
  5. Prompt Assembly
    Retrieved chunks are formatted and ingested as model's context into a prompt template, which also includes the system prompt, the user's original query, and tool-use instructions. 
    MESSAGES = [
        {"role": "system", "content": f"""  {SYSTEM_PROMPT}
                                            # Tool-Use Instructions {'\n\n---\n\n'.join(tool_instructions)}
                                            # Relevant Context {'\n\n---\n\n'.join(top_chunks)}
                                       """
        },
        {"role": "user", "content": user_query}
    ]
  6. Inference
    The complete prompt is sent to the LLM inference provider through the API (e.g., curl). 
    curl https://api.openai.com/v1/chat/completions \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
        "model": "gpt-4o-mini",
        "messages": MESSAGES }' % prompt
  7. Response & Post-Processing
    The model response is streamed or returned, then post-processed (e.g., stripping safety tags) before being displayed to the user. The assistant’s answer is embedded asynchronously (store‑right) in a background thread to avoid blocking the UI. The resulting dense vector is then added to the vector index. 
    def clean_response(text: str) -> str:
    for fn in (strip_openai_tags,
               strip_anthropic_prefixes,
               strip_cohere_stop,
               strip_generic_brackets):
        text = fn(text)
    # Collapse multiple spaces / newlines to a single space for neatness
    return re.sub(r"\s+", " ", text).strip()

    Other common post‑processing steps include:
    [✓] Strip provider‑specific safety tags.
    [✓] Remove or whitelist HTML/markdown that could be unsafe.
    [✓] Extract a < TOOLS > {...} < /TOOLS > JSON block and hand it to the tool dispatcher.
    [✓] Optionally truncate to a token budget.
    [✓] For streaming APIs, apply the same cleaning on each line before echoing to the UI.
